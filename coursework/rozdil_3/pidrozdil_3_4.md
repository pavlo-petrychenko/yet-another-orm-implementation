## 3.4 Драйвер PostgreSQL і компілятор діалекту

Цей підрозділ описує реалізаційний шлях, яким AST-вузли, отримані від `query-builder` та `schema-builder`, перетворюються на конкретний SQL-текст і виконуються СКБД. Архітектурні підстави цього шару закладено у підрозділі 2.3; тут зосереджено увагу на структурі коду: composition root `PostgresDriver`, фабрика компіляторів `PostgresDialect`, спільний контекст `CompilationContext` і два допоміжні сервіси — менеджер параметрів і утиліти ідентифікаторів.

### 3.4.1 `PostgresDriver` як composition root

Клас `PostgresDriver` реалізує інтерфейс `Driver` із п'ятьма основними операціями: `query`, `ddl`, `raw`, `withTransaction` і парою методів життєвого циклу `connect`/`disconnect`. Усередині він агрегує два незалежні компоненти: екземпляр `PostgresDialect` (синглтон, створений у конструкторі) і `PostgresConnection` (`PostgresPoolConnection` за замовчуванням або `PostgresClientConnection`, якщо у конфігурації задано `mode: "client"`). Розділення «що компілювати» і «куди надсилати» утримує драйвер тонким — обидва методи `query` і `ddl` мають однакову форму: викликати відповідний `buildQuery` чи `buildDdl` діалекту, передати `compiled.sql` і `compiled.params` у `connection.query()`, повернути результат у формі `QueryResult`.

Метод `withTransaction` єдиний містить логіку, відсутню у дочірньому `PostgresTransactionalDriver`: він бере виключне користування одним фізичним з'єднанням через `connection.withPinnedClient()`, виконує `BEGIN`, передає у callback щойно створений транзакційний драйвер із `depth = 0` і завершує транзакцію `COMMIT`/`ROLLBACK` відповідно до результату. Pinned-з'єднання утримується на весь час виконання callback, тож усі дочірні запити проходять через ту саму сесію СКБД.

### 3.4.2 Робота з `pg` через шар з'єднань

Шар з'єднань повністю інкапсулює залежність від пакета `pg`. `PostgresPoolConnection` обгортає пул `pg.Pool`, `PostgresClientConnection` — одинокий `pg.Client`. Обидва реалізують той самий інтерфейс `PostgresConnection`, який, окрім простого `query(sql, params)`, оголошує критично важливий метод `withPinnedClient(fn)`: у пулі він виділяє одного клієнта на час виконання callback (через `pool.connect()` і `client.release()` у `finally`), у client-режимі повертає той самий клієнт. Завдяки цьому методу транзакційне з'єднання гарантовано лишається тим самим у всіх дочірніх запитах, що захищає від випадкових ROLLBACK у пулі через перерозподіл клієнтів між запитами однієї транзакції.

### 3.4.3 `PostgresDialect`: 16 компіляторів і єдиний `dispatch`

Клас `PostgresDialect` агрегує шістнадцять компіляторів: одинадцять для DML (по одному на тип запиту — `SELECT`/`INSERT`/`UPDATE`/`DELETE` — та сім для окремих клауз — `Condition`, `Join`, `OrderBy`, `GroupBy`, `Limit`, `Offset`, `Returning`) і п'ять для DDL (`ColumnTypeCompiler`, `CreateTableCompiler`, `AlterTableCompiler`, `DropTableCompiler`, `RenameTableCompiler`). Усі компілятори створюються у момент ініціалізації самого діалекту і живуть протягом усього часу його існування. Це робить діалект безстановим між викликами `buildQuery`/`buildDdl`: жодного приватного стану не накопичується, що дозволяє виконувати компіляцію конкурентно для будь-якої кількості одночасних запитів.

Самі методи `buildQuery(query)` і `buildDdl(query)` мають типовий двоетапний шаблон: створення нового екземпляра `ParameterManager` для зібрання плейсхолдерів, побудова `CompilationContext` для поточної компіляції, а далі — диспетч до відповідного компілятора через `switch (query.type)`. Дискримінований союз `Query`/`DdlQuery` (підрозділи 3.2.1 і 3.3.1) робить цей `switch` повним за варіантами: пропуск нового типу породжує помилку компіляції.

### 3.4.4 `CompilationContext` як механізм передачі залежностей

Усі компілятори приймають другим аргументом об'єкт `CompilationContext` із чотирма полями: `params: ParameterManager`, `utils: DialectUtils`, `compileCondition: (clause) => string` і `compileSelect: (sub: SelectQuery) => string`. Останні два — це замикання, що дозволяють атомарним компіляторам викликати компілятор умов і компілятор `SELECT` (для вкладених підзапитів) без знання конкретних класів. Це і є той механізм, що уможливлює рекурсію в AST: `PostgresConditionCompiler` обробляє `BaseCondition` з правою частиною типу `SelectQuery`, викликаючи `ctx.compileSelect(sub)`, а та компіляція, у свою чергу, через `ctx.compileCondition` може звернутися до тих самих умов знову.

Для DDL-компіляції ці два callback оголошені такими, що кидають помилку: «DDL compilation does not support conditions / nested SELECTs». Така захисна реалізація фіксує контракт явно — DDL не повинен звертатися до DML-частин, — і запобігає випадковому міжмодульному просочуванню.

### 3.4.5 `PostgresParameterManager` і захист від SQL-injection

Параметри SQL-запиту в YAOI ніколи не вбудовуються у текст команди — лише передаються через позиційні плейсхолдери `$1, $2, ...`, що їх драйвер `pg` обробляє через готові prepared statements. Реалізація цього механізму зосереджена у одному простому класі: метод `add(value)` додає значення у внутрішній масив `values` і повертає рядок виду `$N`, де `N` — поточна довжина масиву. Метод `getParams()` віддає накопичений масив у вигляді readonly-структури. Завдяки такій будові ніяка користувацька строка не може потрапити у тіло SQL — навіть escape-методи на ідентифікатори (`PostgresDialectUtils.escapeIdentifier`) перетворюють введені імена через подвоєння подвійних лапок (`"` → `""`) і не довіряють їм як рядкам параметрів.

### 3.4.6 `PostgresSelectCompiler` як демонстрація шаблону

Один із одинадцяти DML-компіляторів — `PostgresSelectCompiler` — демонструє типовий стиль: масив `parts` накопичується послідовно за SQL-граматикою (`SELECT … FROM … JOIN … WHERE … GROUP BY … HAVING … ORDER BY … LIMIT … OFFSET`), кожен опційний фрагмент включається лише за наявності відповідного поля в AST, а делегація до інших компіляторів — клауз-`JOIN`, групування, сортування — виконується через ін'єктовані залежності, передані у конструктор (лістинг 3.3). Підтримку `UNION`/`UNION ALL` реалізовано рекурсивним самовикликом самого `compile()` із вкладеним `SelectQuery` правого операнда.

**Лістинг 3.3 — Кінцевий вихід пайплайна компіляції**

```ts
// AST, побудоване через QueryBuilder (із лістингу 3.1):
const query = qb
  .select({ name: "users" }, { name: "id" }, { name: "email" })
  .where((c) => c
    .where({ name: "is_active" }, "=", true)
    .group(LogicalOperator.AND, (g) => g
      .where({ name: "created_at" }, ">=", new Date("2025-01-01"))
      .orWhere({ name: "is_admin" }, "=", true)
    )
  )
  .orderBy({ name: "created_at" }, OrderDirection.DESC)
  .limit(50)
  .build();

// Виклик через драйвер:
const compiled = dialect.buildQuery(query);
// compiled.sql:
//   SELECT "id", "email" FROM "users"
//   WHERE "is_active" = $1 AND ("created_at" >= $2 OR "is_admin" = $3)
//   ORDER BY "created_at" DESC LIMIT 50
//
// compiled.params: [true, Date("2025-01-01"), true]
```

Ідентифікатори в результуючому SQL завжди беруться у подвійні лапки (через `PostgresDialectUtils.escapeIdentifier`), що дозволяє безпечно використовувати ключові слова СКБД як імена стовпців/таблиць; параметри ж залишаються позиційними плейсхолдерами, що передаються драйверу `pg` окремо.

### 3.4.7 Структура DDL-компіляторів і composite-вирази

DDL-компілятори побудовані за тим самим принципом, що й DML, але мають більш плоску структуру: один `PostgresCreateTableCompiler` оброблює весь `CreateTableQuery`, делегуючи лише компіляцію типу колонки в окремий `PostgresColumnTypeCompiler`. Особливість, варта окремої згадки: `compile()` цього компілятора може повертати не одне, а серію команд, розділених `; ` — основна `CREATE TABLE (...)` плюс окремі `CREATE INDEX` за кожним заданим індексом. Драйвер `pg` коректно обробляє таку multi-statement форму, виконуючи їх послідовно в одному раунді спілкування з сервером.

Цей шар є кінцевим у конвеєрі «AST → SQL → СКБД»; усе, що дотепер описано в розділах 3.1–3.4, складає фундамент, на якому далі будуються модель і два фасади публічного API. Наступний підрозділ переходить до реалізації `BaseModel` — Active Record-частини того подвійного API.
