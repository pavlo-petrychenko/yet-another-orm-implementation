## 3.9 Тестування (Jest + testcontainers)

Тестовий набір YAOI організований навколо трьох взаємодоповнювальних шарів перевірки: юніт-тести Jest, інтеграційні тести з реальною СКБД у одноразовому контейнері та тип-тести з пакетом `expect-type`, що компіляторно перевіряють публічні типи. Такий поділ безпосередньо випливає з вимоги Н.6 (підрозділ 1.6), яка фіксує заборону мокувати реляційну базу даних для інтеграційних сценаріїв і вимагає валідовності типів як окремої форми тестування.

### 3.9.1 Структура й обсяг тестового набору

На момент завершення реалізації набір складається з 87 тестових файлів, поділених між трьома категоріями. Перша — 41 юніт-тест-файл із розширенням `*.spec.ts`, розташовані у теках `__tests__/` поряд із кодом модуля, що тестують; вони охоплюють Query Builder, Schema Builder, окремі компілятори діалекту, побудовники AST і реєстри метаданих. Друга — 46 інтеграційних файлів `*.int.spec.ts` у теках `__integration__/`, що виконують реальні SQL-запити проти PostgreSQL-контейнера. Третя — 8 файлів `*.type-test.ts`, що компілюються разом із юніт-тестами і використовують пакет `expect-type` для статичного твердження поведінки типів. Окремо існують 10 файлів із розширенням `*.test.ts` для специфічних сценаріїв (зокрема, тести CLI), що теж потрапляють під базову Jest-конфігурацію.

### 3.9.2 Дві Jest-конфігурації

Розшарування на швидкі юніт-тести та повільні інтеграційні виконано через дві окремі конфігурації Jest. Файл `jest.config.js` — швидкий потік: глобальний `testMatch` приймає файли `*.spec.ts` і `*.test.ts`, а `testPathIgnorePatterns` явно виключає каталог `/__integration__/`. Такий набір виконується за командою `npm test`, не потребує мережі чи Docker і запускається у локальному ноутбук-режимі за лічені секунди.

Файл `jest.integration.config.js` — повільний потік: `testMatch` обмежений шаблоном `**/__integration__/**/*.int.spec.ts`, додано `globalSetup` і `globalTeardown` із підняттям/зупинкою контейнера, прапорці `testTimeout: 60000` і `maxWorkers: 1`. Останнє — особливо важливе: усі інтеграційні тести виконуються послідовно у одному воркері, тож вони ділять той самий екземпляр PostgreSQL без ризику паралельних змін схеми чи даних. Цей потік запускається окремо командою `npm run test:integration`.

### 3.9.3 `globalSetup`/`globalTeardown` із `@testcontainers/postgresql`

Конкретна реалізація управління контейнером міститься у двох симетричних файлах. `globalSetup.ts` створює нову інстанцію `PostgreSqlContainer("postgres:16-alpine")` з фіксованими реквізитами (`yaoi_it`/`test`/`test`), запускає її через `.start()` і отримує реальні значення `host`, `port`, `user`, `password`, `database` (port — мапований testcontainers у вільний на хості). Далі формується звичайний `pg.Client`, підключається до щойно піднятого контейнера, виконує заздалегідь підготовлений `schema.sql` із декларацією таблиць інтеграційних тестів, і закривається. Реквізити доступу записуються у `process.env.YAOI_IT_PG_*`, через що інтеграційні тести бачать їх через `getEnvConfig()` без передавання прямих посилань на сам контейнер. Сам контейнер прив'язується до `globalThis.yaoiPgContainer` для подальшого використання у teardown.

`globalTeardown.ts` зчитує посилання на контейнер із `globalThis` і викликає `.stop()`. Завдяки тому, що ці методи виконуються Jest'ом у окремих процесах поза тестовими воркерами, контейнер живе протягом усього прогону інтеграційного набору, незалежно від падінь окремих тестів. Якщо процес зупиняється передчасно (наприклад, Ctrl-C), testcontainers має власну infrastructure для finalizer'ів, що зупиняють контейнер по сигналу SIGTERM/SIGINT.

### 3.9.4 `setupDriver` як хелпер фікстур життєвого циклу

Усередині кожного інтеграційного спека життєвий цикл драйвера обслуговується викликом `setupDriver(mode)` із `helpers.ts`. Він реєструє чотири Jest-хука: `beforeAll` створює `pg.Pool` для прямих службових запитів (наприклад, `TRUNCATE`); `afterAll` його закриває; `beforeEach` виконує `TRUNCATE users, orders RESTART IDENTITY CASCADE` і створює свіжий `PostgresDriver` із заданим `mode` (`"pool"` чи `"client"`); `afterEach` дисконнектить його. Повертає фікстура два методи: `getDriver()` для доступу до драйвера зсередини тестів і `rawQuery(sql, params)` для прямих перевірок стану бази через інший пул (важливо: окремий пул запобігає кросс-впливам між службовим і власне тестованим драйвером).

Такий шаблон утримує тіло кожного спека максимально декларативним: тестовий блок виглядає як `it("…", async () => { const driver = fixture.getDriver(); await driver.withTransaction(...); expect(...).toEqual(...) })`, без явного управління підключеннями. Лістинг 3.7 показує реальний фрагмент із набору інтеграційних тестів транзакційного шару.

**Лістинг 3.7 — Інтеграційний тест вкладеної транзакції із savepoint**

```ts
describe("PostgresDriver.withTransaction (integration)", () => {
  const fixture = setupDriver("pool");

  it("nested withTransaction uses savepoints — inner rollback preserves outer writes",
    async () => {
      const driver = fixture.getDriver();
      await driver.withTransaction(async (tx) => {
        await tx.raw("INSERT INTO users (name, email) VALUES ($1, $2)",
                     ["outer", "o@x"]);
        await expect(
          tx.withTransaction(async (sp) => {
            await sp.raw("INSERT INTO users (name, email) VALUES ($1, $2)",
                         ["inner", "i@x"]);
            throw new Error("inner-fail");
          }),
        ).rejects.toThrow("inner-fail");
      });

      // Зовнішня транзакція успішно закомічена (outer), вкладена відкочена (inner)
      const rows = await fixture.rawQuery<{ name: string }>(
        "SELECT name FROM users ORDER BY name");
      expect(rows.map((r) => r.name)).toEqual(["outer"]);
    });
});
```

Цей тест валідує одночасно три механізми, описані у попередніх підрозділах: pinning з'єднання у пулі (підрозділ 3.4), створення savepoint у `PostgresTransactionalDriver` (підрозділ 3.8) і коректний `ROLLBACK TO SAVEPOINT` без скасування зовнішньої транзакції. Подібну форму мають усі 46 інтеграційних спеків — вони перевіряють поведінку через реальні SQL-запити до реальної СКБД, без жодних моків.

### 3.9.5 Тип-тести через `expect-type`

Окрема категорія перевірок — статичні тести, що не виконуються Jest'ом у класичному сенсі, а компілюються разом із кодом і породжують помилки збірки, якщо тип-твердження не виконуються. Кожен такий файл (`*.type-test.ts`) починається з імпорту `expectTypeOf` із пакета `expect-type` і застосовує два типові ідіоми: позитивні твердження виду `expectTypeOf<U4["orders"]>().toEqualTypeOf<Order[]>()` й негативні — у формі коментаря `// @ts-expect-error <пояснення>` перед рядком, де навмисно очікується помилка компілятора. Якщо очікувана помилка не з'являється, сам `@ts-expect-error` піднімає компілятор-`error` (бо коментар «обіцяє» помилку у наступному рядку — порушення цієї обіцянки само по собі є дефектом).

Тип-тести в YAOI охоплюють найскладніші частини публічної системи типів: `Strict<T, A>` із чотирма гілками (narrow без аргументів, із `select`, із `include`, із обома); `ColumnKeys<T>` і `RelationKeys<T>` із виключенням методів; `IncludeConfig<T>` із рекурсивною вкладеністю; `SelectMap<T>` як обмежений до колонок; `Where<T>` із операторами, обмеженими типом стовпця. Завдяки цьому розробник, що зачепить будь-який із цих типів у майбутньому, отримає негайний сигнал від компілятора про порушення семантичних інваріантів — навіть не запускаючи жодного юніт-тесту.

### 3.9.6 Контроль якості коду через лінтер

Третій рубіж захисту — статичний аналіз ESLint, конфігурація якого розглянута у підрозділі 3.1. Серед правил, дотичних саме до тестового коду, варто виокремити три. `no-only-tests/no-only-tests` забороняє `.only` у виразах `describe`/`it`, що блокує характерну категорію помилок «один тест працює локально, але в CI його ніхто не запускає». `@typescript-eslint/no-unused-vars` із префіксом `^_` робить нерозумінні `_unused`-параметри легітимними, не дозволяючи при цьому ховати справді забутий код. `unused-imports/no-unused-imports` автоматично видаляє непотрібні `import`'и під час `lint --fix` — це разом із `consistent-type-imports` (`fixStyle: "inline-type-imports"`) утримує імпорти охайними протягом усього життєвого циклу коду.

### 3.9.7 Зведення тестових гарантій

Описаний тестовий набір валідує три ортогональні властивості реалізації. Юніт-тести (~41 спек) перевіряють чисту логіку без побічних ефектів — будівельників AST, компіляторів діалекту, реєстри метаданих, утиліти; вони виконуються за лічені секунди й покривають більшу частину коду. Інтеграційні тести (46 спеків, `maxWorkers: 1`) підтверджують, що той самий код працює коректно у поєднанні з реальною PostgreSQL-СКБД, що особливо важливо для транзакційного шару, каскадів і `RETURNING`-залежних операцій. Тип-тести (8 файлів) гарантують стабільність публічних типів між рефакторингами ядра.

Цим завершується розділ 3. Описана у підрозділах 3.1–3.9 реалізація — від конфігурації проєкту до тестового набору — складає вхідні дані для розділу 4, у якому розглянуто підсистему міграцій із контролем цілісності та CLI-обгортку, що використовує усі попередні шари (Query Builder, Schema Builder, драйвер, модель і транзакції) як готовий інструментарій.
