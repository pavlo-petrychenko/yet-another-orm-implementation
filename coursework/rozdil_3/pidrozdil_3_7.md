## 3.7 Відношення, eager loading, каскади

Робота зі зв'язками винесена з основної логіки `Repository<T>` у незалежну підсистему `model/internal/`, що поділяється на дві симетричні частини: чотири завантажувачі (по одному на тип зв'язку — `many-to-one`, `one-to-many`, `one-to-one`, `many-to-many`) для читання та каскадний обхід (`walkCascade` + топологічне сортування) для запису. Ця організація дозволяє основному класу `Repository<T>` залишатися компактним і доручати конкретні алгоритми спеціалізованим функціям, кожна з яких живе у власному файлі.

### 3.7.1 `RelationLoader` як стратегія завантаження

Файл `model/internal/relationLoaders/index.ts` оголошує єдиний тип `RelationLoader`:

```text
type RelationLoader = (ds, parents, parentMeta, relation, nestedInclude, depth) => Promise<void>;
```

і реєструє чотири конкретні реалізації у структурі `Record<RelationKind, RelationLoader>`. Така будова — фактично шаблон Strategy на рівні структур даних — дозволяє оркестратору вибрати завантажувач за полем `relation.kind` із метаданих без жодного `switch`-у в основному коді: `const loader = relationLoaders[relation.kind]; await loader(...)`.

### 3.7.2 `loadIncludes` як рекурсивний орхестратор

Функція `loadIncludes(ds, parents, metadata, include, depth)` — точка входу, що викликається з `Repository.find`/`findOne` після основного гідратування. Її алгоритм лінійний: для кожного ключа з об'єкта `include` отримати відповідну `RelationMetadata` (із валідацією «такого зв'язку не існує — `UNKNOWN_RELATION`»), знайти у словнику завантажувач за `relation.kind` і викликати його. Сам завантажувач, у свою чергу, після отримання дочірніх записів за потреби викликає `loadIncludes` рекурсивно, передаючи `depth + 1`.

Глибина рекурсії обмежена константою `MAX_INCLUDE_DEPTH = 10`: при перевищенні піднімається типізована `ModelError` із кодом `"INCLUDE_DEPTH_EXCEEDED"`. Це формальний захист від випадкового рекурсивного включення, у якому система типів `IncludeConfig<T>` залишає лазівку: якщо граф сутностей містить цикл (наприклад, `User.posts.author.posts...`), TypeScript-тип залишається коректним, але рантайм-виконання могло б увійти в нескінченну рекурсію.

### 3.7.3 Спільний шаблон «fetch by FK IN (...)» і запобігання N+1

Усі чотири завантажувачі побудовані за одним принципом, що повністю усуває проблему N+1: для всього масиву батьківських сутностей виконується один-єдиний SQL-запит виду `SELECT … FROM children WHERE fk IN ($1, $2, …, $k)`. Внутрішня утиліта `fetchByColumnIn(ds, target, meta, columnName, ids)` (модуль `relationLoaders/shared.ts`) інкапсулює цей патерн: збирає `QueryBuilder().select(...).where(whereIn(fk, ids))`, виконує через `ds.getDriver().query(...)` і гідратує результат. Завдяки цьому навіть глибоко вкладені `include` дають кількість SQL-запитів, рівну сумарній кількості зв'язків у конфігурації, а не кількості рядків (для `2` батьків × `3` зв'язки = 4 запити: один на батьків + по одному на кожен зв'язок).

### 3.7.4 Чотири конкретні завантажувачі

`loadManyToOne` найпряміший: збирає значення FK-стовпця у власників, виконує `fetchByColumnIn(target, referencedColumn, ids)` і заповнює `parent[relation] = byKey.get(fkValue)` через попередньо побудовану `Map<unknown, object>`. `loadOneToMany` симетричний: збирає значення PK у батьків, виконує `fetchByColumnIn(target, inverseFk, parentIds)` і групує результати через `groupByKey(...)` — `parent[relation] = group(parentPk) ?? []` для кожного батька. `loadOneToOne` — інтелектуальна обгортка: якщо `relation.joinColumn` визначений (тобто FK на «своєму» боці), делегує у `loadManyToOne`; інакше використовує ту саму внутрішню функцію, що й `loadOneToMany` (`fetchChildrenByInverseFk`), але повертає лише перший елемент групи замість усього масиву.

`loadManyToMany` — найскладніший. Він виконує два послідовні SQL-запити. Перший — `SELECT (parentFk, targetFk) FROM joinTable WHERE parentFk IN (...)` — повертає пари ключів зі зв'язкової таблиці. Другий — `fetchByColumnIn(target, targetPk, targetIds)` — підвантажує самі цільові сутності за списком унікальних `targetFk`, зібраних із першого запиту. Після цього у пам'яті будується дві мапи (`childByPk` та `grouped`), що поєднують пари зі зв'язкової таблиці з фактичними об'єктами цільової сутності. Такий двоетапний підхід дозволяє повторно використати ту саму `fetchByColumnIn` без розгалуження SQL-логіки і коректно обробити випадки, коли один цільовий об'єкт належить кільком батькам одночасно.

Лістинг 3.6 показує, як саме поведе себе `loadIncludes` для типового вкладеного запиту.

**Лістинг 3.6 — Eager loading із вкладеним `include` і його SQL-слід**

```ts
const users = await User.find({
  where: { isActive: true },
  include: {
    posts: { include: { comments: true } },
    profile: true,
  },
});

// Послідовність SQL-запитів, що буде виконано
// (для прикладу: знайдено 3 активних користувача з id=1,2,3):
//
//  1. SELECT * FROM "users" WHERE "is_active" = $1                 -- $1 = true
//  2. SELECT * FROM "posts" WHERE "user_id" IN ($1,$2,$3)          -- $* = 1,2,3
//  3. SELECT * FROM "profiles" WHERE "user_id" IN ($1,$2,$3)
//  4. SELECT * FROM "comments" WHERE "post_id" IN ($1,$2,…)        -- усі знайдені posts
//
// Усього: 4 SELECT-запити незалежно від кількості користувачів,
// дописів і коментарів.
```

### 3.7.5 Каскадне збереження: двофазний алгоритм

Каскади на боці запису побудовані інакше від читання. Виявлення необхідності каскадного обходу виконується предикатом `hasCascadeChildren(entity, metadata)`: він проходить усі зв'язки у метаданих із прапорцем `cascade: true` і повертає `true`, якщо хоча б один із них містить значення (не `undefined`, не `null` і не порожній масив). Викликається ця функція у `Repository.save` і `Repository.insert`; у випадку позитивної відповіді операція автоматично обгортається у `resolveDriver().withTransaction(...)`, а решта роботи доручена `walkCascade(root, metadata, ds, tx, mode)`.

Реалізація `walkCascade` поділена на два кроки. Перший — `buildCascadePlan(root, metadata, ds)`, що повертає `CascadePlan` із двома масивами: топологічно впорядкованим списком вузлів `order: CascadeNode[]` і списком зв'язків `m2mLinks: M2MLinkPlan[]`. Сортування виконується ітеративно за принципом «спершу те, що нікому не залежить»: батьки `many-to-one` йдуть перед своїми власниками, а власники `one-to-many` — перед своїми дочірніми. Кожен вузол може містити масив `fkBackfills` — описи FK-полів, що мають бути проставлені на цьому вузлі після персистенції джерела (наприклад, `post.user_id` потрапляє з `user.id` після `INSERT INTO users RETURNING id`). Другий крок — цикл `for (const node of plan.order)`: спершу `applyFkBackfills(node)` копіює значення з уже персистованих джерел у поточний вузол, далі `persistNode(node, ds, tx, mode)` викликає `Repository.saveWithDriver` або `Repository.insertWithDriver` залежно від режиму та наявності значень первинного ключа.

### 3.7.6 Зв'язкові таблиці `many-to-many` як окрема фаза

`many-to-many`-зв'язки оброблюються поза основним топологічним порядком. Причина — рядки зв'язкової таблиці потребують одночасної присутності обох ключів, тож вони не вкладаються у послідовну схему «спершу батько, потім дитина». Усі такі зв'язки зібрано окремим списком `m2mLinks` під час побудови плану і виконуються після завершення основного циклу — функцією `persistRelationLinks(plan.m2mLinks, tx)` через прямий `INSERT INTO joinTable (joinCol, inverseJoinCol) VALUES (...)`.

Усе разом це утворює послідовну й передбачувану схему: незалежно від форми вхідної сутності, операція виконується в одній транзакції, у правильному порядку, з автоматичним поширенням первинних ключів між батьками та дітьми. Розробник у прикладному коді бачить лише виклик `await user.save()` із попередньо заповненим `user.posts = [...]` і `user.profile = {...}` — решту вирішує підсистема каскадів.

Описана модель завантаження і збереження зв'язків доповнює загальну архітектуру моделі та завершує опис ядра доступу до даних. Наступний підрозділ переходить до реалізації транзакційного шару, без якого більшість описаних механізмів — каскади, тестова ізоляція, ambient-tx — не функціонувала б.
