## 3.5 BaseModel (Active Record)

Реалізація Active Record-фасаду YAOI зосереджена у одному абстрактному класі `BaseModel`, що становить близько 180 рядків коду і повністю покладається на `Repository<T>` як спільне ядро (підрозділ 2.5). Цей підрозділ описує те, як саме статичні та інстанс-методи `BaseModel` зберігають типобезпеку через поліморфне `this`, як організована резолюція `DataSource` через prototype chain і як `BaseModel` автоматично долучається до ambient-транзакції.

### 3.5.1 Поліморфний `this` як механізм статичної типобезпеки

Найвиразніший прийом реалізації — це використання `this`-параметра у сигнатурах статичних методів: кожен такий метод оголошений із параметром `<This extends typeof BaseModel>`, після чого параметри й тип повернення формулюються через `InstanceType<This>`. На прикладі `find`: коли користувач викликає `User.find(args)`, TypeScript розв'язує `This` як саме клас `User`, тож `InstanceType<This>` стає `User`, а параметр `args` обмежений до `FindArgs<User>`. Якщо клас-нащадок `BaseModel` має додаткові поля, недоступні у самому `BaseModel`, ця конструкція гарантує, що користувач отримає коректний тип результату без явних анотацій. Той самий шаблон застосовано до всіх десяти статичних методів — `findOne`, `findOneOrFail`, `find`, `count`, `exists`, `create`, `insert`, `insertMany`, `saveMany`, `deleteMany`, `upsert`.

Для `find`/`findOne`/`findOneOrFail` сигнатура додатково містить умовний тип на основі прапорця `narrow: true`: коли користувач його задає, результат звужується до `Strict<InstanceType<This>, A>` — точної форми, обчисленої з аргументів `select`/`include` (підрозділ 2.6); коли не задає — повертається повний тип сутності. Така конструкція дозволяє розробнику свідомо обирати між простотою (`User.find()` повертає `User[]`) і максимально вузькою формою (`User.find({ narrow: true, select: { id: true } })` повертає `Pick<User, "id">[]`).

### 3.5.2 Резолюція `DataSource` через prototype chain

Кожен виклик статичного методу починається із пошуку `DataSource`, який обслуговує даний клас сутності. Реалізація використовує `WeakMap<AnyClassRef, DataSource>` із приватним ім'ям `overrides`, у який потрапляють асоціації, встановлені через `BaseModel.useDataSource(ds)`. Сама резолюція виконується у функції `resolveDataSource(cls)`: вона починає з самого класу і піднімається prototype chain через `Object.getPrototypeOf`, повертаючи перше знайдене значення; якщо ланцюг доходить до `Function.prototype` без збігу, вмикається fallback на глобальний синглтон `getDataSource()` із `model/dataSourceRegistry.ts`.

Семантика такої побудови важлива для практичного використання. По-перше, `useDataSource` на батьківському класі автоматично успадковується нащадками — це дозволяє оголосити «свій» базовий клас для проєкту й конфігурувати `DataSource` один раз, без повторення на кожній сутності. По-друге, конкретна сутність може мати власний override, що перекриває батьківський — використовується у тестах чи мультидатабазних застосунках. По-третє, виключно глобальний синглтон (без жодного override) — найшвидший шлях для скриптів і простих CLI-сценаріїв, де явне керування `DataSource` зайве.

### 3.5.3 Інтеграція з ambient-транзакцією

Внутрішня функція `getRepoFor(cls)` об'єднує резолюцію `DataSource` із консультацією у ambient-контексті. Спершу резолвиться `DataSource`, потім викликається `ambientEntityManagerFor(ds)`: якщо у async-стеку зараз активна транзакція цього `DataSource`, повертається `EntityManager`, з якого і береться репозиторій із вже встановленим `txDriver`; якщо ні — використовується звичайний `ds.getRepository(cls)`. Прикладний код у вигляді `User.find(...)` або `user.save()` працює однаково всередині й поза транзакцією — а вибір між pinned-tx драйвером і пулом виконує одна функція ядра.

### 3.5.4 Інстанс-методи: `save`, `delete`, `reload`, `loadRelation`

Інстанс-частина `BaseModel` містить чотири методи, що працюють із поточним об'єктом. `save` делегує виконання у `Repository.save(this)` — реалізація останнього сама вирішує, виконати `INSERT` чи `UPDATE`, залежно від наявності первинного ключа. `delete` і `reload` спершу обчислюють `Where`-фільтр за значеннями первинних ключів через приватну функцію `pkWhereOf(instance, metadata)`, що читає метадані з декорованих первинних колонок (підрозділ 3.4 розділу 2) і збирає об'єкт виду `{ id: 42 }` або складніший за композитного PK. `loadRelation` приймає ключ зв'язку (тип ключа обмежений `keyof this`, тобто перевіряється на стадії компіляції) і делегує у `Repository.loadRelation`.

Функція `pkWhereOf` піднімає типізовану `ModelError` із кодом `"NO_PRIMARY_KEY"` у двох випадках: коли сутність взагалі не має оголошеного первинного ключа (виявляється на момент компіляції метаданих) і коли первинний ключ присутній, але його значення `undefined` чи `null` на інстансі — типова помилка при спробі видалити чи оновити свіжостворений об'єкт, що ще не пройшов `INSERT`. Лістинг 3.4 ілюструє декларацію сутності, що успадковується від `BaseModel`, і кілька викликів його API в одному дусі.

**Лістинг 3.4 — Active Record-шлях через `BaseModel`**

```ts
@Entity({ tableName: "users" })
class User extends BaseModel {
  @PrimaryKey()
  public id!: number;

  @Column()
  public email!: string;

  @Column({ name: "is_active" })
  public isActive!: boolean;

  @OneToMany(() => Post, (post) => post.author)
  public posts!: Relation<Post[]>;
}

setDataSource(ds);  // глобальний DataSource на весь застосунок

// Статичний шлях: автоматично резолвить DataSource, типи виводяться без анотацій
const active = await User.find({ where: { isActive: true } });
const exists = await User.exists({ email: "alice@example.com" });

// Інстанс-шлях: працює з конкретним об'єктом, делегує у Repository
const user = User.create({ email: "bob@example.com", isActive: true });
await user.save();         // INSERT
user.isActive = false;
await user.save();         // UPDATE
await user.reload();       // повторне читання з БД
await user.loadRelation("posts");  // ключ перевірений на стадії компіляції
await user.delete();       // DELETE WHERE id = user.id
```

### 3.5.5 Делегація як прийом утримання компактності

Жоден зі статичних чи інстанс-методів `BaseModel` не містить власної логіки доступу до даних — усі вони складаються з одного-двох рядків: отримати `Repository<T>` через `getRepoFor`, прокинути аргументи. Лише три приватні допоміжні функції (`resolveDataSource`, `getRepoFor`, `pkWhereOf`) утримують орієнтовно тридцять рядків коду; решта 150 рядків — це сигнатури з аналізом типів, де компілятор виконує усю роботу зі звуження. Така будова реалізує принцип, сформульований у підрозділі 2.5: AR і Repository поділяють спільне ядро, а різниця між ними — суто у формі публічного API, що бачить розробник.

Як саме реалізовано це спільне ядро — `Repository<T>` — а також механізми, що уможливлюють кастомні репозиторії та DI-інтеграцію, розглянуто у наступному підрозділі.
