# ВИСНОВКИ

У роботі спроєктовано, реалізовано та обґрунтовано ефективність типобезпечної ORM-бібліотеки YAOI (Yet Another ORM Implementation) для мови TypeScript із підтримкою декількох SQL-діалектів, подвійним публічним API і послідовним порівняльним аналізом з існуючими рішеннями. Поставлену у вступі мету досягнуто, усі вісім сформульованих завдань виконано, що підтверджується змістом шести розділів пояснювальної записки, повним вихідним кодом бібліотеки та демонстраційним застосунком.

Перші два завдання — аналіз сучасного стану TypeScript-ORM і формування вимог до типобезпечної бібліотеки — виконано у першому розділі. На матеріалі TypeORM, Objection.js і Drizzle виявлено три повторювані компроміси цих рішень: часткова типобезпека через рядкові ідентифікатори стовпців і шаблонні літерали, фрагментація ідіом API і непослідовна підтримка декількох SQL-діалектів. На основі виявлених прогалин сформульовано шість функціональних і шість нефункціональних вимог до YAOI, ключовими з яких є наскрізна статична типобезпека без `any` у публічних типах ядра, шарова архітектура з ациклічним графом залежностей і розділення конвеєрів DML та DDL на рівні системи типів. Перелічено чотири критерії успіху, що формально визначили межу між «реалізовано» і «потребує доопрацювання» для подальших розділів.

Завдання з третього по шосте — проєктування наскрізного типобезпечного конвеєра, розробка подвійного публічного інтерфейсу, реалізація ядра з восьми модулів і побудова підсистеми міграцій з CLI — становлять серцевину роботи й виконані у другому, третьому й четвертому розділах. У другому розділі сформовано модульну декомпозицію бібліотеки з ациклічним графом залежностей `metadata → query-builder / schema-builder → drivers → model / migrations → cli` і обґрунтовано вибір ключових архітектурних патернів: декоратор-орієнтовану декларацію сутностей, окремі двох-пайплайнові QueryBuilder і SchemaBuilder для розмежування DML і DDL, двосторонній фасад моделі для подвійного API і `AsyncLocalStorage`-контекст для прозорого ambient-передавання транзакції. У третьому розділі ці рішення матеріалізовано у вихідному коді: реалізовано шістнадцять компіляторів запитів із чітко окресленими інтерфейсами, типобезпечні `Where<T>`, `IncludeConfig<T>` і `Strict<T, A>`-типи, що виводять форму результату безпосередньо з форми запиту, обидва стилі публічного API на спільному ядрі `Repository<T>` без дублювання логіки доступу до даних. У четвертому розділі побудовано підсистему міграцій з контролем цілісності за допомогою SHA-256-контрольних сум, рекомендаційного блокування PostgreSQL з фіксованим ключем `0x59414f49` для безпечного конкурентного запуску, транзакційного виконання DDL і CLI-інтерфейсу з командами `migrate:make`, `migrate:up`, `migrate:down`, `migrate:status`. Покриття інтеграційних тестів на реальній PostgreSQL, розгорнутій через testcontainers, безпосередньо валідує поведінку всіх перелічених підсистем.

Сьоме й восьме завдання — порівняльний аналіз з конкурентними бібліотеками і розробка демонстраційного застосунку — закривають практичну валідацію проєкту. У п'ятому розділі виконано систематичне зіставлення YAOI з TypeORM, Objection.js і Drizzle за п'ятьма критеріями — типобезпека, API і developer experience, подвійний API та DI-сценарії, архітектура й розширюваність, підхід до міграцій. Підсумкова таблиця 5.3 фіксує найвищу оцінку YAOI за вісьмома з дванадцятьох критеріїв і ділення першості за двома, з єдиним змістовним компромісом — кількістю фактично підтриманих діалектів (один проти 6-8 у конкурентів) — що, проте, компенсується найменшою вартістю архітектурного додавання нового діалекту. У шостому розділі реалізовано демонстраційний застосунок «mini-shop» на базі Fastify з чотирма сутностями, що покриває всі заявлені сценарії — handwritten-міграції, поєднання Active Record і кастомного Repository у єдиному кодовому стилі, narrow-typed запити з трирівневим `include`, транзакційне оформлення замовлення з ambient-передаванням контексту і повноцінне docker-compose-стенд із postgres, разовими міграціями, seed'ом і API-сервером. На рівні шостого розділу зафіксовано також практичний зворотний зв'язок щодо developer experience, що формує реалістичний перелік виявлених обмежень.

За ступенем впровадження результати роботи доведено до стану відкритої програмної бібліотеки: вихідний код YAOI публікується на платформі GitHub, скомпільована версія розміщується у реєстрі пакетів npm під ім'ям `@iriskaik/yaoi`, демонстраційний застосунок постачається у складі репозиторію в підкаталозі `examples/mini-shop` з docker-compose-orchestration-конфігурацією, що дозволяє відтворити повний стенд однією командою. Можливі сфери застосування — серверні Node.js-застосунки, що потребують типобезпечного доступу до PostgreSQL: внутрішньокорпоративні API-сервіси, бекенди вебзастосунків, мікросервісні архітектури зі строгими вимогами до контролю міграційного стану у багатоінстансному деплойменті, де властивості SHA-256-checksum-валідації та advisory-lock-серіалізації стартів є особливо цінними.

Науково-технічна значущість роботи полягає у трьох ключових внесках. По-перше, у системі типів TypeScript розгорнуто наскрізну типобезпеку «end-to-end» від запиту до результату через тип `Strict<T, A>`, що виводить форму повернення безпосередньо з форми запиту й елімінує клас runtime-помилок, типових для рядкових ORM-API. По-друге, реалізовано подвійний публічний інтерфейс (Active Record + Repository) як два рівноправні фасади над спільним ядром `Repository<T>` — конфігурація, не виявлена у жодному з трьох розглянутих опонентів і така, що знімає типову для індустрії «архітектурну дихотомію» вибору стилю на старті проєкту. По-третє, у міграційній підсистемі поєднано три захисні механізми — SHA-256-контроль цілісності, rollback verification і PostgreSQL advisory lock — у конфігурацію, що формально гарантує лінеаризовність застосування міграцій і виявлення несанкціонованих модифікацій застосованих файлів, властивість, не реалізована у жодній з трьох порівняних бібліотек.

Доцільність продовження досліджень убачається у п'яти конкретних напрямах, кожен з яких природно лягає у спадкоємність прийнятих архітектурних рішень. Першочерговим є реалізація компіляторів для MySQL і SQLite, що завершить перехід від «архітектурно мультидіалектної, наразі однодіалектної» бібліотеки до повноцінної мультидіалектної. Друге — окреме методологічно правильне дослідження продуктивності з вимірюванням latency і throughput на типових сценаріях у порівнянні з конкурентами. Третє — `yaoi schema:pull`-команда для інтроспекції наявної схеми бази даних і генерації відповідних декорованих сутностей. Четверте — `yaoi migrate:generate`-команда для діфф-генерації міграцій з порівняння декларації сутностей і поточної схеми. П'яте — офіційні інтеграційні пакети для популярних фреймворків (NestJS, Fastify, Hono), що формалізують патерн підключення YAOI як шар над наявним `DataSource` і `repositoryRegistry`. Розгорнутий перелік перспектив, включно з менш пріоритетними пунктами (хуки життєвого циклу сутності, event-system на рівні DataSource), наведено у підрозділі 5.7.3 пояснювальної записки.
