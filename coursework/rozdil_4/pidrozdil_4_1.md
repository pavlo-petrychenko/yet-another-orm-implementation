## 4.1 `MigrationRunner`, таблиця `yaoi_migrations`, статуси

Підсистема міграцій YAOI реалізована як окремий модуль `src/migrations/` із власною моделлю даних, контрактом файлу міграції та публічним класом `MigrationRunner`. Цей підрозділ описує дві опорні точки модуля — таблицю-журнал застосованих міграцій та сам контракт файлу-міграції, — а також типи статусів, що повертаються командою `migrate:status`.

### 4.1.1 Файл міграції як модуль із двома функціями

Кожна міграція в YAOI — це окремий TypeScript-файл, що експортує об'єкт за замовчуванням, що відповідає інтерфейсу `Migration`:

```ts
interface Migration {
  up(schema: SchemaBuilder): Promise<void>;
  down(schema: SchemaBuilder): Promise<void>;
}
```

Обидві функції отримують екземпляр `SchemaBuilder` (підрозділ 3.3), прив'язаний до драйвера поточної транзакції. Усі DDL-операції — `createTable`, `alterTable`, `dropTable`, `renameTable`, `raw` — виконуються через цей `SchemaBuilder`, що гарантує однорідність API між схемою, керованою міграціями, і схемою, керованою прикладним кодом. Приклад типової міграції наведений у лістингу 4.1.

**Лістинг 4.1 — Файл міграції зі створенням таблиці**

```ts
import type { Migration, SchemaBuilder } from "yaoi";

const migration: Migration = {
  async up(schema: SchemaBuilder): Promise<void> {
    await schema.createTable("posts", (t) => {
      t.id();
      t.integer("user_id").notNull().references("users").onDelete("CASCADE");
      t.string("title", 255).notNull();
      t.text("body").notNull();
      t.boolean("is_published").notNull().default(false);
      t.timestamps();
      t.index(["user_id"], { name: "idx_posts_user" });
    });
  },

  async down(schema: SchemaBuilder): Promise<void> {
    await schema.dropTable("posts");
  },
};

export default migration;
```

Завантаження файлу виконується через функцію `loadMigrationFile`: вона викликає динамічний `await import(absolutePath)`, читає `default`-експорт (з fallback на весь модуль, якщо `default` не визначений) і перевіряє, що обидві властивості `up` і `down` є функціями. Невідповідність піднімає `InvalidMigrationFileError` із посиланням на конкретний шлях — користувач негайно бачить, який саме файл не відповідає контракту.

### 4.1.2 Discovery: сканування директорії

Discovery виконується функцією `discoverMigrations(dir, extensions)` із модуля `loader.ts`. Алгоритм лінійний: `fs.readdir(dir, { withFileTypes: true })` повертає список елементів директорії; відфільтровуються лише регулярні файли з розширеннями зі списку (`[".ts", ".js"]` за замовчуванням), додатково виключаються `.d.ts`, `.spec.`, `.test.` патерни; результат сортується лексикографічно за іменем файлу. Завдяки сортуванню за іменем YAOI покладається на префіксну схему іменування — про неї детальніше у підрозділі 4.5.

Кожен елемент результату — об'єкт `DiscoveredMigration` із трьох полів: `name` (ім'я без розширення), `absolutePath` і lazy-метод `load()`, що при першому виклику виконає динамічний імпорт і повернеться валідований `Migration`. Лінива загрузка важлива з двох причин: по-перше, вона дозволяє виявити drift через чексуми (підрозділ 4.2) без виконання коду міграції; по-друге, помилка у синтаксисі однієї з міграцій не блокує виконання решти команди.

### 4.1.3 Таблиця-журнал `yaoi_migrations`

Журнал застосованих міграцій зберігається у самій базі даних, у таблиці зі стандартним іменем `yaoi_migrations` (налаштовується через `tableName` у конфігурації). Її схема оголошується самим YAOI через свій же `SchemaBuilder` у функції `ensureTrackingTable(driver, tableName)` — це і є самозастосування ORM до власної інфраструктури:

```ts
await schema.createTable(tableName, (t) => {
  t.ifNotExists();
  t.id();
  t.text("name").notNull().unique();
  t.text("checksum").notNull();
  t.timestamp("applied_at", { withTimezone: true })
    .notNull()
    .defaultRaw("NOW()");
});
```

Прапорець `ifNotExists()` робить виклик ідемпотентним: повторне виконання `bootstrap()` не порушує існуючої таблиці. Унікальний індекс на `name` запобігає випадковому подвійному записуванню тієї самої міграції — навіть якщо у логіці runner'а виявиться race condition (захищена advisory lock'ом — підрозділ 4.4), вторинна спроба буде відкинута БД.

### 4.1.4 Чотири класи статусу і команда `migrate:status`

Метод `status()` повертає масив `MigrationStatus[]` з шести полів на кожен запис: `name`, `applied`, `appliedAt`, `storedChecksum`, `fileChecksum`, `mismatch`. Алгоритм формує об'єднання двох множин — імена файлів з директорії та імена з таблиці — і для кожного імені окремо обчислює статус. Це породжує чотири взаємно виключні класи запису:

1. **applied** — файл існує, рядок у таблиці існує, чексуми збігаються (нормальний стан);
2. **pending** — файл існує, рядка у таблиці немає (готова до застосування);
3. **mismatch** — файл існує, рядок існує, чексуми різні (drift — детально у підрозділі 4.2);
4. **orphan** — рядок є, файл відсутній (`fileChecksum === null`); типовий випадок видаленого файлу.

Команда `migrate:status` форматує цей масив у компактну таблицю, що зручно читається у терміналі: ім'я, статус, час застосування, перші 12 символів чексуму. Розробник отримує повне уявлення про стан міграційної історії одним викликом.

Як саме обчислюються чексуми та як виявляється drift у вже застосованих міграціях — розглянуто далі.
