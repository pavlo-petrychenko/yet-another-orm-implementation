## 4.4 Advisory lock для concurrency safety

Третя гарантія підсистеми міграцій захищає від типового сценарію розподілених систем: одночасний запуск `migrate:up` із кількох інстансів додатку. Без додаткового захисту такі запуски утворюють класичну гонку — обидва runner'и одночасно читають журнал, обидва бачать одну й ту саму pending-міграцію, обидва намагаються її застосувати, у результаті у БД виконується подвійний DDL і вторинний `INSERT` падає з порушенням `UNIQUE`-обмеження на `yaoi_migrations.name` із напіввиконаним станом. YAOI вирішує цю проблему через PostgreSQL advisory locks.

### 4.4.1 `pg_advisory_xact_lock` із фіксованим ключем

Реалізація захисту зосереджена в одній функції — `acquireRunnerLock(driver)`:

```ts
const LOCK_KEY = 0x59414f49;  // ASCII "YAOI"

export async function acquireRunnerLock(driver: Driver): Promise<void> {
  await driver.raw("SELECT pg_advisory_xact_lock($1)", [LOCK_KEY]);
}
```

PostgreSQL надає механізм advisory locks через групу функцій `pg_advisory_*`, що блокують за довільним числовим ключем без зв'язку з реальними рядками чи таблицями. Серед цих функцій `pg_advisory_xact_lock(key)` спеціально розрахована на використання всередині транзакції: лок утримується до кінця поточної транзакції і автоматично звільняється при `COMMIT` або `ROLLBACK` — додавати окремий `pg_advisory_unlock` не потрібно. Це робить її простою для коректного використання: неможливо «забути звільнити» лок, бо звільнення відбувається разом із завершенням транзакції.

Числовий ключ `0x59414f49` — це ASCII-кодування рядка «YAOI» (0x59='Y', 0x41='A', 0x4F='O', 0x49='I'). Він глобальний для самого продукту YAOI і не залежить від конкретної бази чи схеми; це означає, що advisory locks YAOI ізольовані від інших додатків, що користуються PostgreSQL advisory locks за іншими ключами, але серіалізуються між собою — два процеси, що запускають YAOI-міграції одночасно проти тієї самої бази, обов'язково чекатимуть один на одного.

### 4.4.2 Розміщення локу всередині кожної транзакції

`acquireRunnerLock(tx)` викликається у трьох місцях `MigrationRunner`: у `bootstrap()` (перед `ensureTrackingTable`), у `up()` (перед перевіркою/виконанням кожної pending-міграції) та у `down()` (перед `migration.down`). У всіх трьох випадках виклик — перша інструкція всередині `withTransaction(async (tx) => { … })`, тож порядок дій такий: `BEGIN` → `SELECT pg_advisory_xact_lock(YAOI)` → решта логіки → `COMMIT`/`ROLLBACK`. Завдяки тому, що `pg_advisory_xact_lock` блокує синхронно (запит «висить», доки інший процес тримає лок), вторинний runner просто чекатиме на тому ж рядку SQL, поки перший не завершиться. Тільки після цього він зможе зчитати журнал, побачити, що міграцію вже застосовано, і тихо пропустити її.

### 4.4.3 Idempotent re-check всередині критичної секції

Навіть із advisory lock'ом залишається тонкий випадок: між початком виконання `up()` (де читається журнал) і `BEGIN` для конкретної міграції могла відбутися зміна стану — інший runner, що тримав лок раніше, вже міг застосувати ту саму міграцію. Тому всередині критичної секції виконується додатковий `readApplied(tx, tableName)` із перевіркою `if (insideRows.some((r) => r.name === file.name)) return false`. Якщо рядок виявлено — runner тихо повертає `false` і `up()` записує `didApplyThis = false` без додавання у `newlyApplied`. Це робить runner повністю ідемпотентним: повторні запуски на одному й тому ж стані БД не виконують жодних дій і не повідомляють про помилку.

### 4.4.4 Чому advisory lock, а не table lock

Альтернативний підхід — використати `LOCK TABLE yaoi_migrations IN EXCLUSIVE MODE` — теж серіалізував би runner'и, але має два недоліки. По-перше, табличний лок захищає лише власне таблицю-журнал, тоді як критична секція runner'а охоплює DDL над зовсім іншими таблицями застосунку; advisory lock із фіксованим ключем закриває весь застосунок як одну логічну операцію. По-друге, табличний лок взаємодіє з повсякденними запитами до самої таблиці — якщо діагностичний код у застосунку випадково читає `SELECT * FROM yaoi_migrations`, він блокується разом із runner'ом, що псує його як індикатор стану. Advisory lock же повністю невидимий для звичайних SQL-запитів, тож не має таких побічних ефектів.

Описана конструкція — транзакційність + чексуми + advisory lock — становить технічне ядро підсистеми міграцій. Залишається останній компонент: CLI-обгортка, що робить це ядро доступним зі стандартних команд оболонки. Її реалізація розглядається у заключному підрозділі розділу.
