## 6.4 Оцінка зручності використання

Реалізація mini-shop виконана у позиції стороннього розробника, що використовує YAOI як бібліотеку, без знання внутрішньої архітектури за межами публічного API. Цей розділ підсумовує спостереження за такою користувацькою діяльністю — те, що працювало природно, і те, що виявило обмеження, потенційно вартих усунення.

### 6.4.1 Що показало себе зручним

**Подвійний API без перенесення коду.** Можливість почати з AR-стилю на простіших endpoint'ах і додати кастомний `OrderRepository` лише там, де є транзакційна логіка — без правок існуючих маршрутів — є цінною властивістю на практиці. У TypeORM розробник часто вимушений приймати рішення «AR чи Repository» наперед на рівні всього проєкту через нерівнозначну підтримку; у YAOI ця двозначність відсутня.

**Ambient-tx як вирішення прокидування контексту.** Те, що `OrderRepository.placeOrder` всередині `dataSource.transaction(...)` отримує `makeRepository(Product)` і `makeRepository(OrderItem)` без явного передавання `EntityManager`, виявилося особливо помітним при написанні: не довелося ускладнювати сигнатури методів параметром-контекстом. Альтернативний шлях (передавати `EntityManager` явно через всі шари) є джерелом помилок, бо легко забути передати в один з вкладених викликів, і операція мовчки виконається поза транзакцією. У YAOI такого класу помилки конструктивно неможливі.

**Типобезпека в `where`/`include`.** Під час кодування `routes/orders.ts` IDE автодоповнював ключі `where: { ... }` і структуру `include: { items: { include: { product: ... } } }`, що зменшило час пошуку дефіниції і виключило типографічні помилки. Усвідомлено зроблено спробу написати `include: { foo: true }` — TypeScript негайно повідомив `Object literal may only specify known properties, and 'foo' does not exist in type 'IncludeConfig<Order>'`.

**CLI-команди як один npm-bin.** Інтеграція з Fastify не потребувала жодних ORM-специфічних wrapper'ів. `npm run migrate:up` працює як з хосту, так і з контейнера, оскільки `yaoi`-bin поставляється самим пакетом і не вимагає глобальної установки.

### 6.4.2 Що виявилося незручним

**Ручне зазначення `type` у `@Column`.** TypeScript на рівні Stage 3-декораторів не передає в декоратор інформацію про статичний тип властивості; YAOI вимагає, щоб тип колонки був явно зазначений у `@Column({ type: "string" })`. Для рутинних полів типу `email: string` це дає очевидний бойлерплейт, який міг би виводитися автоматично, якби декоратор використав `emitDecoratorMetadata` (`reflect-metadata`-схема, до якої вдаються TypeORM/Mikro-ORM). YAOI свідомо відмовився від цього шляху на користь Stage 3 (підрозділ 3.3), але ціна — у форматі додаткових `type:`-параметрів. Це нормальний компроміс, але варто фіксувати як виявлене обмеження.

**Хардкодний union `ModelErrorCode`.** Спроба використати `ModelError` для доменних кодів `"INSUFFICIENT_STOCK"`/`"PRODUCT_INACTIVE"` спричинила помилку компілятора, бо `ModelErrorCode` — фіксований union, у якому таких літералів немає. Прикладний код мусив завести власний `DomainError`-клас із паралельною таксономією. Зручніший варіант — параметризувати клас власним union-типом (`ModelError<Codes extends string>`), що дав би розширюваність без втрати типобезпеки.

**Відсутність хуків життєвого циклу.** Поле `createdAt: Date` у `User`, `Product`, `Order` довелося явно заповнювати у `seed.ts` (`createdAt: new Date()`) і в `placeOrder` спадково через дефолтне значення міграції (`defaultRaw("NOW()")`). Хук `@BeforeInsert` був би прозорішим способом виставити це значення для всіх трьох сутностей у одному місці. Це обмеження вже зафіксовано як пункт 2 у переліку 5.7.2.

**Розкид defaults між декларацією та міграцією.** Поле `Product.isActive` має дефолт `true` у міграції (`t.boolean("is_active").notNull().default(true)`), але не у декларації `@Column`. При використанні `Product.insert({ sku, name, priceCents, stock })` без `isActive` поле буде заповнено базою. Це працює, але стан defaults розкиданий між двома місцями. Натуральніше було б, якби `@Column({ default: true })` синхронізувалося і з runtime-INSERT'ом, і з генерацією DDL (що вимагає codegen-міграцій, теж зафіксовано у 5.7.3).

**Відсутність валідації на рівні ORM.** YAOI приймає будь-який `Partial<T>` у `insert`/`save` без runtime-перевірок. У реалізації `placeOrder` довелося власноруч перевіряти `quantity > 0`, `product.isActive`, `product.stock >= line.quantity`. Це нормально для бібліотеки з мінімальною поверхнею, але натяк на майбутній напрям — інтеграційні пакети (`@yaoi/zod`, наприклад), що дозволили б декларувати схему валідації поряд із сутністю.

### 6.4.3 Розподіл AR vs Repository: коли який природніший

На основі написання шести endpoint'ів складається тверда емпірична межа:

- **AR-стиль** природний, коли вся операція над сутністю описується одним викликом (`Product.find`, `User.findOne`, `Product.count`) і не потребує спільної транзакційної інваріанти з іншою сутністю.
- **Repository-стиль** (із підкласом) природний, коли операція потребує: (а) атомарної взаємодії з кількома сутностями, (б) додаткової бізнес-логіки (валідація стану, обчислення сум), (в) семантичного імені, яке точніше описує дію, ніж генеричні `find`/`save` (`placeOrder` чи `findRecentByUser`).

Жоден з шести endpoint'ів mini-shop не вимагав «змішування» обох стилів у межах однієї HTTP-операції, що свідчить про чисту ортогональність вибору. Це підтверджує тезу підрозділу 5.4.1 про обидва стилі як рівноправні фасади.

### 6.4.4 Висновки розділу

Реалізація демонстраційного застосунку підтверджує, що архітектурні рішення YAOI з розділів 2-4 практично робочі: типобезпека спрацьовує в реальному коді, транзакції з ambient-tx елімінують поширений клас помилок, міграційний CLI бездоганно інтегрується у контейнерну оркестрацію. Виявлені обмеження (ручне зазначення `type`, фіксований `ModelErrorCode`, розкид defaults, відсутність хуків і валідації) є реалістичним зворотним зв'язком, що визначає напрями подальшого розвитку, переліченими у підрозділі 5.7.3, і утверджує позиціонування YAOI як ORM «для типового 80% сценаріїв» — з усвідомленням того, що залишок 20% покривається кастомним кодом на прикладному рівні без втрати ясності.

З цим розділом завершено детальне опрацювання теоретичної і практичної частин роботи. Загальні висновки за усією роботою сформульовано у заключній частині.
