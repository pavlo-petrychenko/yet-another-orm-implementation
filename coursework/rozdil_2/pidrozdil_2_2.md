## 2.2 Розділення DML і DDL пайплайнів на рівні типів

Поділ операцій доступу до бази на маніпуляцію даними (DML) і визначення схеми (DDL) на рівні типів закріплює вимогу Н.3 та запобігає випадковому змішуванню двох конвеєрів у спільному коді. Структурно це означає, що від білдера до драйвера обидві категорії запитів подорожують власними типами без спільного супертипу; проміжні значення одного конвеєра не можуть бути передані у точку входу іншого без помилки компіляції.

### 2.2.1 Дві паралельні AST-сім'ї

Кожен конвеєр представлений власним дискримінованим юніоном з окремим переліком тегів-типів (лістинг 2.1).

**Лістинг 2.1 — AST-сім'ї DML і DDL**

```ts
// DML
export enum QueryType {
  SELECT = "SELECT",
  INSERT = "INSERT",
  UPDATE = "UPDATE",
  DELETE = "DELETE",
}
export type Query =
  | SelectQuery
  | InsertQuery
  | UpdateQuery
  | DeleteQuery;

// DDL
export enum DdlQueryType {
  CREATE_TABLE = "CREATE_TABLE",
  ALTER_TABLE  = "ALTER_TABLE",
  DROP_TABLE   = "DROP_TABLE",
  RENAME_TABLE = "RENAME_TABLE",
}
export type DdlQuery =
  | CreateTableQuery
  | AlterTableQuery
  | DropTableQuery
  | RenameTableQuery;
```

Юніони `Query` і `DdlQuery` не мають спільної надмножини у відкритому API. Спільним залишається лише номінальне поняття «таблиця» (`TableDescription`) — реквізит, що використовується обома гілками для опису цілі операції.

### 2.2.2 Два окремих контракти доступу

Розділення поширюється вгору по графу залежностей на інтерфейси `Driver` та `Dialect` (лістинг 2.2). Драйвер експонує дві дискримінаційні вхідні точки: `query(...)` для DML і `ddl(...)` для DDL; діалект — дві симетричні точки компіляції: `buildQuery(...)` та `buildDdl(...)`.

**Лістинг 2.2 — Сигнатури входів драйвера й діалекту**

```ts
export interface Driver {
  query<TRow>(q: Query): Promise<QueryResult<TRow>>;
  ddl(q: DdlQuery): Promise<QueryResult>;
  // ...
}

export interface Dialect {
  buildQuery(q: Query): CompiledQuery;
  buildDdl(q: DdlQuery): CompiledQuery;
  // ...
}
```

Перевага такого подвоєння — статичне відкидання неприпустимих комбінацій ще на стадії компіляції: спроба передати `CreateTableQuery` у `driver.query(...)` (або `SelectQuery` у `driver.ddl(...)`) є помилкою типу і не доходить до рантайму. Додатково тип результату звужений: DML-точка повертає `QueryResult<TRow>` із рядками шуканої форми, тоді як DDL-точка повертає `QueryResult` без рядків. Реалізація діалекту посилює це рішення на рівні внутрішнього контексту компіляції: при обробці DDL-вузла спроби викликати компілятори умов чи вкладеного `SELECT` гарантовано кидають помилку, унеможливлюючи появу DML-конструкцій усередині DDL-AST.

### 2.2.3 Білдер як джерело AST

Жоден з білдерів — ні з модуля `query-builder`, ні з модуля `schema-builder` — не повертає рядок SQL. Кінцева точка білдера — це AST-вузол (`SelectQuery`, `CreateTableQuery` тощо), який передається у драйвер; перетворення AST у SQL виконує діалект, і це єдине місце системи, де з'являється текст запиту. Механіку цього перетворення та структуру діалекту як набору компіляторів розглянуто у підрозділі 2.3.

### 2.2.4 Наслідки розділення

Розділення дає чотири практичні наслідки, що випливають із побудови:

- **Статична несумісність гілок.** Випадкове змішування DML- і DDL-конструкцій у спільному фрагменті коду фіксується компілятором без потреби в рантайм-перевірках.
- **Незалежне тестування конвеєрів.** Компілятори DML і DDL перевіряються ізольовано — на рівні «вхід: AST-вузол, вихід: текст SQL та параметри» — без піднятої бази даних.
- **Безпечніша еволюція схеми.** Підсистема міграцій працює виключно з DDL-AST, що позбавляє її прямого доступу до DML-операцій над даними поза межами заявленої схеми.
- **Точка розширення для діалектів.** Додавання нового діалекту вимагає реалізації двох сімей компіляторів (DML і DDL) у новому класі `Dialect`, без жодних змін у білдерах чи у шарі моделі.

Сам шар драйверів і діалектів, що використовує ці контракти, розглянуто у наступному підрозділі.
