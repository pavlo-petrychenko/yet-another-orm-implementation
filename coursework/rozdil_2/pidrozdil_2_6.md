## 2.6 Стратегія типобезпеки

Вимога Н.1 формулює мету у негативних термінах — відсутність `any` у публічних типах ядра, — але архітектурно вона реалізується системою позитивних правил: кожен аргумент високорівневих методів має бути обмежений до тих ключів сутності, які мають сенс для відповідної операції, а тип результату має звужуватися автоматично без надмірних анотацій із боку розробника. Стратегія наслідує підхід Drizzle (підрозділ 1.4.3) із звуженням типу на кожному кроці DSL, але переносить його на високорівневий `find`-API замість запитного builder'а. Завдяки цьому розробник зберігає ергономіку Active Record (1.2.1), не втрачаючи статичних гарантій повноти проєкцій і коректності операторів.

### 2.6.1 Branded `Relation<T>` і похідні множини ключів

Без службового розрізнення між скалярними полями та полями-зв'язками жоден із похідних типів не може коректно реагувати на форму конкретної сутності. Тип `Relation<T>` оголошений як перетин `T & { readonly [RELATION_BRAND]: T }` із `unique symbol`-брендом, що існує лише в системі типів і не відображається у рантаймі. Завдяки цьому поле сутності, оголошене як `posts: Relation<Post[]>`, формально несе ту саму поверхню, що й `Post[]` — звертатися до нього у прикладному коді можна тим самим способом, — але типи `Post[]` і `Relation<Post[]>` стають розрізнюваними структурно.

На цій основі побудовано три похідні множини ключів. `RelationKeys<T>` відбирає ключі, типи яких сумісні з `Relation<unknown>`. `ColumnKeys<T>` — її доповнення, з окремим виключенням методів. `ScalarKeys<T>` — підмножина `ColumnKeys<T>`, чий тип сумісний зі `ScalarParam` — тобто такий, що може бути значенням-параметром у `WHERE`. Усі три обчислюються mapped-типами з умовним фільтром і слугують атомами всіх вищих типів.

### 2.6.2 `Where<T>` і `OrderBy<T>`

Тип `Where<T>` обмежений саме `ScalarKeys<T>`, що автоматично виключає relations і методи з допустимих ключів предикатів. Кожен ключ-значення може бути або скаляром (рівність), або об'єктом `WhereOperators<V>` із набором операторів — `$eq`, `$ne`, `$gt`, `$lt`, `$gte`, `$lte`, `$in`, `$nin`, `$like`, `$ilike`, `$isNull`. Доступність рядкових операторів додатково обмежена умовним типом `V extends string ? string : never`, тож виклик `where: { age: { $like: "%" } }` для числового стовпця спричиняє помилку компіляції, а не рантайму. Логічні з'єднувачі `$and`, `$or`, `$not` оголошені рекурсивно на тому самому `Where<T>`, що дозволяє довільне вкладення без падіння типобезпеки.

Тип `OrderBy<T>` має просту форму — масив об'єктів вигляду `{ [K in ScalarKeys<T>]?: "asc" | "desc" }` — і успадковує те саме обмеження множини ключів.

### 2.6.3 Рекурсивний `IncludeConfig<T>` для eager-joins довільної глибини

Для опису вкладеного завантаження зв'язків YAOI використовує рекурсивний тип `IncludeConfig<T>`, у якому ключі обмежені `RelationKeys<T>`, а значення — `true` (завантажити сам зв'язок) або `{ include?: IncludeConfig<RelationTarget<T[K]>> }` (завантажити зв'язок плюс конфігурацію вкладених включень). Аргумент `RelationTarget<T[K]>` обчислюється окремою кондиційною формулою, що знімає бренд `Relation<...>` і розгортає масив до елемента, тож у вкладеному `include` ключі знову обмежені зв'язками цільової сутності. Завдяки такій самовикличній побудові тип залишається коректним на будь-якій глибині графа.

### 2.6.4 `Strict<T, A>` як виведення форми результату

Центральний тип `Strict<T, A>` зводить разом усі попередні: він приймає аргументи виклику `A` і виводить точну форму повернення. Його обчислення складається з двох гілок, що перетинаються. Перша обробляє `select`: за наявності ключа `select` форма результату звужується до `Pick<T, Extract<keyof S, ColumnKeys<T>>>` — лише вибрані стовпці; за його відсутності використовується `Omit<T, RelationKeys<T>>` — повна сутність без relations за замовчуванням. Друга гілка обробляє `include`: для кожного включеного зв'язку рекурсивно обчислюється `Strict<U, { include: NI }>`, де `U` — тип-ціль relation, а `NI` — вкладена конфігурація. Тип `IncludedRelations<T, I>` коректно враховує масивну форму зв'язку один-до-багатьох та опційну форму один-до-одного.

Сигнатура `find<A extends FindArgs<T>>(args?: A)` повертає `Array<A extends { narrow: true } ? Strict<T, A> : T>`. Прапорець `narrow: true` тут є opt-in інструментом: коли розробникові достатньо звичайного `T[]`, він не активується; коли потрібен максимально вузький тип результату — додає одне зайве поле в аргументах. Лістинг 2.6 показує, як це виглядає з боку користувача API.

**Лістинг 2.6 — Типобезпечний `find` з виведенням форми результату**

```ts
const users = await User.find({
  narrow: true,
  where: { isActive: true, email: { $like: "%@example.com" } },
  select: { id: true, email: true },
  include: { posts: { include: { comments: true } } },
});
// Тип users виводиться компілятором без явних анотацій як:
//   Array<
//     Pick<User, "id" | "email"> & {
//       posts: Array<Omit<Post, "comments" | "author"> & {
//         comments: Array<Omit<Comment, "post" | "author">>;
//       }>;
//     }
//   >

// Характерні помилки виявляються на стадії компіляції:
User.find({ where: { foo: 1 } });
//                   ~~~ "foo" не існує у User
User.find({ where: { email: { $gt: 5 } } });
//                                   ~ "number" не присвоюється "string"
User.find({ include: { isActive: true } });
//                     ~~~~~~~~ "isActive" не зв'язок
User.find({ select: { posts: true } });
//                    ~~~~~ "posts" не стовпець
```

### 2.6.5 Trade-off

Платою за таку наскрізну типобезпеку є складність внутрішніх типів: `Strict<T, A>` та `IncludedRelations<T, I>` побудовані як ланцюги кондиційних і mapped-типів із вкладеною рекурсією, що при супроводі вимагає від розробників ядра достатнього досвіду в системі типів TypeScript. Із боку прикладного коду ця складність не виявляється — більшу її частину компілятор виводить автоматично, а розробникові залишається спостерігати лише змістовне повідомлення про помилку. Цей компроміс прийнятий усвідомлено: складність винесено в один модуль (`model/types`), і її поглинання дає чисте DX для всіх інших шарів — від AR-фасаду до Repository та `EntityManager`.

Питання передачі транзакційного контексту, що не охоплюється описаною статичною системою типів, розв'язано окремим механізмом ambient-пропагації, який розглянуто у наступному підрозділі.
