## 5.5 Архітектура та розширюваність (вартість нового діалекту)

Підтримка кількох SQL-діалектів — критерій, за яким архітектура ORM перевіряється на реальну гнучкість. Цей підрозділ оцінює, що означає «додати новий діалект» у кожній з чотирьох бібліотек, як цей шлях організований архітектурно і яка приблизна вартість роботи.

### 5.5.1 YAOI: явна точка розширення, один діалект із чотирьох передбачених

Архітектура YAOI явно виносить діалектні особливості в інтерфейс `Dialect` із двома методами (`buildQuery(query)` і `buildDdl(query)`) і шістнадцять імплементацій `*Compiler`-класів, що оголошені на рівні `common/compilers/` як абстрактні інтерфейси (підрозділ 3.4). Підтримка нового діалекту реалізується створенням нового набору з шістнадцяти компіляторів плюс окремий `*ParameterManager` (для діалектних плейсхолдерів: `$N` у PostgreSQL, `?` у MySQL/SQLite) і `*DialectUtils` (для діалектного escape ідентифікаторів: подвійні лапки у PostgreSQL, backtick у MySQL, подвійні лапки у SQLite). Орієнтовний обсяг для нового діалекту — близько 700-900 рядків коду в одному модулі `src/drivers/mysql/` чи `src/drivers/sqlite/`, без зміни жодного рядка в спільному `src/drivers/common/`, `src/query-builder/` чи `src/model/`. Перелік `DBType` уже містить трьохзначення (`POSTGRES`, `MYSQL`, `SQLITE`), `DriverFactory` готовий до них через `switch (config.type)`, а нереалізовані гілки кидають `NotImplementedError` із описом, який саме діалект ще не підтримано. Тобто YAOI наразі підтримує один діалект (PostgreSQL), але архітектурно повністю готовий до розширення.

### 5.5.2 TypeORM: вбудована підтримка багатьох діалектів, монолітний код driver'а

TypeORM історично підтримує найширший набір діалектів серед опонентів: PostgreSQL, MySQL, MariaDB, SQLite, MS SQL Server, Oracle, CockroachDB, SAP HANA. Архітектурно ці діалекти живуть у `src/driver/`-теки з підкаталогами `postgres`, `mysql`, `sqlite` тощо, кожен з яких реалізує клас типу `PostgresQueryRunner`/`MysqlQueryRunner`. Однак спільний код у TypeORM значно більш переплетений: окремі діалектні гілки часто містять умовні `if (this.driver instanceof PostgresDriver) { ... } else if (this.driver instanceof MysqlDriver) { ... }` всередині спільних класів `QueryBuilder` і `EntityManager`, що утворює щільну сітку cross-references. Додавання нового діалекту тут — задача рівня форка проєкту, а не локальна зміна; на практиці нові діалекти потрапляють у бібліотеку через PR від мейнтейнерів, а не через user-land розширення.

### 5.5.3 Objection.js: повне делегування Knex.js

Objection.js не має власного шару діалекту — він повністю покладається на Knex.js, що підтримує PostgreSQL, MySQL, MariaDB, SQLite, MS SQL, Oracle через однотипний інтерфейс `Knex.Client`. Це означає, що додавання нового діалекту у Objection.js — задача рівня Knex, а не самого Objection.js, і виконується одноразово у репозиторії `knex/knex`. З погляду розробника, що використовує Objection.js, перехід з PostgreSQL на MySQL — це здебільшого зміна `knexfile.js` без зміни бізнес-коду; з погляду архітектури бібліотеки — це повністю «вирішена ззовні» задача, що складно віднести до сильної чи слабкої сторони самого Objection.js.

### 5.5.4 Drizzle: окремі підпакети на діалект із паралельними API

Drizzle обрав радикально протилежну до Knex стратегію: кожен діалект — окремий npm-підпакет із власним API. Сутність-таблиця оголошується через `pgTable` для PostgreSQL, `mysqlTable` для MySQL, `sqliteTable` для SQLite; набори типів колонок різні (`serial` у PG vs `int` із `autoIncrement` у MySQL); навіть `db.select(...)` повертає різні класи `PgSelect`, `MySqlSelect`, `SQLiteSelect`. Така будова дає максимальну типобезпеку діалектно-специфічних функцій (наприклад, `jsonb_set` у PostgreSQL компілятор знає; в MySQL — ні), але має ціну: код, написаний для одного діалекту, не портується на інший без перепису. Це антипод TypeORM-стратегії: там абстракція максимальна, тут — мінімальна.

### 5.5.5 Порівняння за «вартістю» нового діалекту

Сумарна оцінка вартості додавання діалекту наведена у таблиці 5.1.

**Таблиця 5.1 — Архітектурна вартість підтримки нового SQL-діалекту**

| Бібліотека    | Точка розширення                  | Кількість файлів  | Обсяг (LOC) | Хто розширює                |
|---------------|-----------------------------------|-------------------|-------------|-----------------------------|
| YAOI          | `src/drivers/<name>/` (16 компіляторів + Driver + Connection) | ~20 | ~700-900 | user-land можливе         |
| TypeORM       | `src/driver/<name>/` + поправки у спільних класах | ~30+ | ~3000+ | переважно core team       |
| Objection.js  | (через Knex) — окремий проєкт | n/a | n/a | через PR у knex/knex     |
| Drizzle       | окремий npm-підпакет `@drizzle-orm/<name>` | ~50+ | ~5000+ | core team               |

YAOI має найменший обсяг роботи на новий діалект завдяки ясній сегментації коду по 16 компіляторам із чітко окресленими інтерфейсами. TypeORM при більшому обсязі коду вимагає глибокого знання внутрішньої архітектури через cross-references у спільних класах. Objection.js делегує задачу Knex, що з одного боку — економія, з іншого — за межами власної кодової бази. Drizzle обрав path-of-least-coupling за рахунок повного дублювання діалектних API, тож вартість найбільша, але типобезпека діалектно-специфічних функцій — найвища.

### 5.5.6 Висновок

YAOI поступається TypeORM і Objection.js у фактичній наявності діалектів на сьогоднішній день — PostgreSQL проти 6-8 діалектів у конкурентів. Однак за критерієм «вартість додавання нового діалекту» YAOI у вигіднішій позиції завдяки явній архітектурній сегментації; це означає, що при наявності розробницького ресурсу підтримка MySQL чи SQLite досяжна локально, без перевідкриття загальної архітектури. Сама ця властивість — «архітектурно мультидіалектна, наразі однодіалектна» — корелює з молодістю проєкту і входить до переліку обмежень YAOI, явно зазначених у підрозділі 5.7.
