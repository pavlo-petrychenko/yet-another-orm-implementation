## 1.5 Зведена порівняльна таблиця функціоналу

Спостереження з підрозділу 1.4 зведено у таблиці 1.2 за сімома формалізованими критеріями, що покривають три ключові осі компромісів: архітектурну (як описуються сутності та як будуються запити), типобезпекову (межі статичної верифікації) та інфраструктурну (підтримка діалектів, міграцій і транзакцій). YAOI у зіставленні не наведено — її проєктовані характеристики систематизуються у підрозділі 1.6 як перелік вимог, а пряме порівняння з опонентами на однакових сценаріях виконується у розділі 5.

**Таблиця 1.2 — Зіставлення TypeORM, Objection.js та Drizzle за функціональними критеріями**

| Критерій | TypeORM | Objection.js | Drizzle |
|---|---|---|---|
| Архітектурний патерн | Active Record і Data Mapper на вибір розробника | Active Record-подібний клас моделі над Query Builder | Query-first без класичного AR або Data Mapper |
| Декларація сутностей | Класи з декораторами `@Entity`, `@Column`, метадані через `reflect-metadata` | Класи-нащадки `Model` плюс окрема JSON Schema | Об'єкти-значення TypeScript (`pgTable(...)`) як єдине джерело істини |
| Стратегія типобезпеки | Декораторна; у складних `QueryBuilder`-сценаріях типи фактично деградують до `any` | Часткова; багато ключових інтерфейсів (`where`, `select`) приймають довільні рядки | Повне виведення з опису схеми; тип результату звужується на кожному кроці DSL |
| Мультидіалектність | PostgreSQL, MySQL, MariaDB, SQLite, Microsoft SQL Server, Oracle Database | Усі діалекти, що підтримує Knex.js (PostgreSQL, MySQL, SQLite та інші) | PostgreSQL, MySQL, SQLite через окремі субмодулі з різними API |
| Міграції | Вбудована підсистема та власний CLI | Зовнішні — делегуються Knex.js | Окремий пакет `drizzle-kit` із генерацією за різницею схем |
| Eager loading | Параметр `relations: { ... }` у методах пошуку | Виразний DSL `withGraphFetched`/`withGraphJoined` для вкладених графів | Окремого механізму немає, виконується явними `JOIN`-ами у запиті |
| Транзакції | `DataSource.transaction(...)`, вкладені через savepoints | `transaction()` Knex.js, вкладені через savepoints | `db.transaction(...)`, вкладені через savepoints |

Із зіставлення видно, що жодне з трьох рішень не поєднує одночасно ергономіки Active Record для типових CRUD-сценаріїв, повної типобезпеки результату запитів (зокрема для проєкцій і вкладених завантажень) та явної мультидіалектної архітектури з єдиним механізмом міграцій. TypeORM купує ергономіку коштом точності типів у складних запитах; Objection.js — широку діалектну базу через Knex.js коштом статичних гарантій; Drizzle — повноту виведення типів коштом відсутності об'єктно-моделювальних патернів. Виявлений розрив безпосередньо мотивує перелік вимог до YAOI, формалізованих у наступному підрозділі.
