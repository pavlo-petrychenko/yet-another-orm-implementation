## 1.3 Типобезпека в ORM та межі статичної верифікації

**Типобезпекою ORM** у цій роботі названо властивість, за якої максимум помилок, пов'язаних із доступом до бази, виявляється на етапі компіляції. Для ORM її можна розкласти на чотири виміри: коректність імен стовпців у `WHERE`/`SELECT`/`ORDER BY`, відповідність типу параметра типу стовпця, точний тип результату для проєкцій і `Pick`-подібних форм, а також автоматичне виведення форми відповіді для вкладених `include`. Лістинг 1.1 ілюструє різницю між сирим і типізованим стилями.

**Лістинг 1.1 — Нетипобезпечний та типобезпечний SELECT**

```ts
// (а) рядковий шаблон — друкарська помилка проявиться лише у рантаймі
const rows = await pool.query(
  `SELECT * FROM users WHERE emial = $1`, [email],
);

// (б) генерик-параметризовано — ім'я та тип перевіряються компілятором
const users = await orm.from(User)
  .where({ email })
  .select(['id', 'name'])      // тип результату: Pick<User, 'id'|'name'>[]
  .execute();
```

Засобами TypeScript [1] для досягнення такого ефекту є генерики з обмеженнями, типи-літерали, оператори `keyof`, `Pick`, `Omit` та умовні типи, що дозволяють вивести тип повернення з форми вхідних аргументів без рантайм-перевірок.

### 1.3.1 Три підходи до досягнення типобезпеки

**Schema-first з генерацією коду.** Канонічним джерелом істини є опис на DSL або інтроспекція бази; окремий генератор перетворює його у файли з типами. Дає найжорсткіший контроль відповідності типів реальній схемі, але вимагає окремого кроку збірки та залежить від інструмента-генератора.

**Декораторний підхід.** Сутність описується класом TypeScript із декораторами, метадані зчитуються через `reflect-metadata` або статично. Ергономічний і добре поєднується з впровадженням залежностей (DI), але залежить від експериментальних можливостей мови, має рантайм-видатки та може розсинхронізуватися з реальною схемою без міграційної дисципліни.

**Query-first із виведенням типів зі значень.** Схема описується як значення TypeScript, тип сутності виводиться з них генериками. Дає максимально виразну типобезпеку без зовнішніх інструментів, але вимагає глибоких знань системи типів і погано вписується в патерн Active Record.

### 1.3.2 Межі статичної верифікації

Жодна система типів не покриває таких аспектів:

- **стан бази** (існування рядка, унікальність) — завжди рантайм-факт;
- **сирий SQL** у функціях `raw('...')` непрозорий для компілятора;
- **розсинхронізація** TypeScript-декларацій із фактичною схемою БД — питання дисципліни міграцій;
- **поліморфні зв'язки, CTE, віконні функції** на практиці вимагають часткового відступу до сирого SQL.

Отже, метою при проєктуванні нової бібліотеки є не абсолютизація типобезпеки, а її локалізація: масові випадки помилок мають ловитися компілятором, а решта — концентруватися у чітко визначених «крайових» точках API. Цей принцип покладено в основу архітектурних рішень YAOI, розглянутих у розділі 2.
