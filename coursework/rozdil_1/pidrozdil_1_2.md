## 1.2 Архітектурні патерни персистентності

Більшість сучасних ORM, незалежно від мови та платформи, побудована на каталозі сталих архітектурних рішень, які М. Фаулер у праці [8] назвав патернами шару доступу до даних. Нижче розглянуто п'ять патернів, що мають визначальне значення для подальшого аналізу: Active Record, Data Mapper, Repository, Query Builder та Unit of Work. Опис кожного складається з трьох частин — суть, переваги і недоліки, типові сфери застосування. Конкретні бібліотеки, що реалізують той чи інший патерн, систематично розглянуто у підрозділі 1.4.

### 1.2.1 Active Record

**Суть.** Active Record (AR) ототожнює сутність доменної моделі з рядком таблиці: клас містить поля, що відповідають стовпцям, і методи доступу до даних (`save`, `update`, `delete`, `findById`, `findAll` тощо) безпосередньо як власні методи сутності. Інстанс одночасно несе бізнес-стан і знає, як себе зберегти у базі. Зазвичай патерн доповнюється статичними методами класу для пошуку та масових операцій.

**Переваги.** Мінімальна кількість шарів і шаблонного коду; інтуїтивний інтерфейс для типових CRUD-сценаріїв; швидкий старт для невеликих доменів і прототипів; природне відображення концепції «сутність» на одну сутність мови програмування.

**Недоліки.** Жорстке змішування бізнес-логіки із логікою доступу до даних ускладнює модульне тестування (сутність неможливо інстанціювати без зв'язку з базою); важко вводити різні джерела даних для одного типу; патерн погано масштабується у складних доменах, де бізнес-логіка переростає рамки однієї таблиці.

**Сфери застосування.** Бекенди з домінуючими CRUD-сценаріями; адміністративні панелі; невеликі сервіси й прототипи; навчальні проєкти, де простота важливіша за чистоту шарів.

### 1.2.2 Data Mapper

**Суть.** Data Mapper розводить доменну модель і логіку її персистентності у незалежні шари: доменний клас не містить жодних згадок про базу, а окремий клас-маппер відповідає за двостороннє перетворення «об'єкт ↔ рядок таблиці». Сутність не знає, як себе зберегти; ця відповідальність винесена в інфраструктуру.

**Переваги.** Чисте розділення доменного та інфраструктурного шарів; домен можна розробляти й тестувати без бази даних; можливість переключати джерела даних або змінювати схему без впливу на бізнес-логіку; легше досягати відповідності принципу єдиної відповідальності.

**Недоліки.** Значно більший обсяг шаблонного коду; необхідність ручної підтримки маперів або складніший рантайм-механізм для їх генерації; складніша конфігурація проєкту, особливо для невеликих доменів, де переваги патерну не встигають проявитися.

**Сфери застосування.** Складні доменні моделі з насиченою бізнес-логікою (DDD-стиль); системи з очікуваною заміною джерела даних; великі команди, де ізоляція шарів дозволяє паралельну розробку.

### 1.2.3 Repository

**Суть.** Repository надає доменному коду колекційно-подібний інтерфейс для отримання та збереження сутностей, приховуючи запити та маппінг за абстракцією на зразок `Repository<User>` із методами `findById`, `findAll`, `save`, `remove`, `findBy(criteria)`. На відміну від Data Mapper, який є технічним перетворювачем, Repository — це доменна абстракція над колекцією об'єктів певного типу.

**Переваги.** Висока тестованість через підстановку in-memory реалізацій; природна сумісність із впровадженням залежностей (DI), оскільки сервіси приймають інтерфейс Repository у конструкторі; інкапсуляція складних запитів у методах із бізнесово зрозумілими іменами; легка зміна стратегії зберігання без впливу на доменну логіку.

**Недоліки.** Потребує інфраструктури впровадження залежностей або принаймні явної передачі залежностей; ризик «розпухання» інтерфейсу при додаванні все нових спеціалізованих методів; додатковий шар абстракції, який у простих CRUD-сценаріях не дає виграшу.

**Сфери застосування.** Сервіс-орієнтовані застосунки із чітким відокремленням бізнес-логіки; проєкти, що використовують підходи DDD або Clean Architecture; кодові бази з високими вимогами до модульного тестування.

### 1.2.4 Query Builder

**Суть.** Query Builder надає програмний інтерфейс для побудови SQL-виразів через послідовність викликів методів замість конкатенації рядків. Виклики на зразок `.select(...).from(...).where(...).join(...).orderBy(...).limit(...)` поступово накопичують абстрактне синтаксичне дерево, яке у кінцевій точці серіалізується у валідний SQL під обраний діалект.

**Переваги.** Усунення цілого класу помилок, пов'язаних із ручною конкатенацією SQL і SQL-ін'єкціями; композиційність — фрагменти запитів можна виносити в окремі функції й перевикористовувати; контрольоване перенесення між діалектами за рахунок шару серіалізації; за умови добре розвиненої системи типів — значна доля статичної верифікації запитів.

**Недоліки.** Власний DSL потрібно вивчати окремо від SQL; повна типобезпека досягається лише за рахунок складних структурних типів і потребує продуманого проєктування генериків; продуктивність нижча, ніж у заздалегідь скомпільованого SQL, через накладні витрати на побудову та серіалізацію.

**Сфери застосування.** Реалізація обох попередніх патернів (AR і Data Mapper) зазвичай спирається на внутрішній Query Builder; також він часто пропонується кінцевому користувачеві як «спадковий» шар для нестандартних запитів, які важко виразити через високорівневий API.

### 1.2.5 Unit of Work

**Суть.** Unit of Work відстежує множину об'єктів, змінених у межах однієї бізнес-операції, і координує запис цих змін у базу як єдиної транзакційної одиниці. Зазвичай UoW веде три внутрішні множини — «нові», «змінені» та «видалені» об'єкти — і на момент завершення транзакції виконує відповідні DML-операції в коректному порядку з урахуванням залежностей між сутностями.

**Переваги.** Атомарність бізнес-операцій із гарантією узгодженості; оптимізація кількості звернень до бази шляхом пакетування записів; точка для реалізації оптимістичного блокування та виявлення конфліктів; основа для роботи з ідентичністю об'єктів (Identity Map) у межах однієї сесії.

**Недоліки.** Значна реалізаційна складність — потрібен механізм виявлення змін (dirty tracking), управління ідентичністю об'єктів і коректний порядок операцій із врахуванням зовнішніх ключів; ризик неявних побічних ефектів у точці завершення транзакції, що ускладнює налагодження.

**Сфери застосування.** ORM, що позиціонуються як повноцінні рантайми персистентності (на зразок Hibernate чи MikroORM); складні бізнес-сценарії, де одна команда оновлює узгоджено десятки сутностей.

### 1.2.6 Узагальнення

Розглянуті патерни не є взаємовиключними: реальна ORM зазвичай поєднує кілька з них, обираючи між AR та Data Mapper як основу для публічного API, доповнюючи його Query Builder для нестандартних запитів і Repository як рекомендованою стратегією інтеграції з бізнес-кодом, та реалізуючи Unit of Work на рівні сесії або транзакції. У підрозділі 1.4 показано, які саме комбінації обирають TypeORM, Objection.js і Drizzle, а у розділі 2 — як підхід «подвійного API» YAOI поєднує AR і Repository на одному ядрі.
